# 2026-07 Deep Codebase Audit and Remediation Roadmap

**Audit date:** 2026-07-09
**Baseline reviewed:** `origin/main` at `3952c602` before this remediation branch
**Scope:** React/Vite application, Supabase migrations/RLS/RPCs, all Edge Functions, storage access, tests/CI, PWA delivery, dependencies, bundle output, and operational configuration.
**Method:** repository-wide static scans, authorization tracing, migration-policy review, dependency/governance checks, test/build execution, and manual data-flow inspection. This is an implementation backlog, not a claim that every item is already fixed.

## Executive assessment

The application has a solid recent governance layer and a clean app typecheck, but it still carries the signature of rapid AI-assisted growth: duplicated privileged code, old permissive policies, many warning-only quality failures, inconsistent trust boundaries, and broad unmeasured test risk. The biggest risk is not a single vulnerable component; it is that adjacent copies of a pattern drift and bypass later safety improvements.

The first remediation PR addresses the highest-confidence active issues:

- PDF Edge Functions now authenticate approved roles, accept only this project's Storage URLs, bound source fetches, reject redirects and oversized/non-PDF inputs, avoid logging signed URLs, and write random non-overwriting objects with a fixed one-hour URL lifetime.
- `create-transport-request` now verifies the caller, ignores privilege-escalating `requested_by` values, validates sub-rental/job/department linkage, performs data mutations as the caller so RLS remains authoritative, and compensates for failed item creation.
- `sub_rentals` receives a corrective RLS migration and dynamic pgtap coverage because its prior policies compared aliases to themselves (for example, `p.department = p.department`), unintentionally granting broad authenticated access.

These are high-value reductions, not a conclusion to the audit. The phases below deliberately separate urgent containment from structural work so that shipping security patches does not become an unbounded refactor.

### P1 — close exposed trust-boundary and operational gaps (next 2–4 weeks)

| ID | Finding and impact | Remediation plan | Exit criteria / evidence |
| --- | --- | --- | --- |
| SEC-06 | `security-audit` can accept unbounded/high-rate traffic and falls back to an event-supplied user id when identity resolution fails. This permits event pollution and unnecessary storage pressure. | Apply the existing shared bounded-body and rate-limit primitives; require verified identity for authenticated events; give public reports a separate anonymous schema/retention path; deduplicate client error fingerprints. | Load test confirms rate limiting; forged `user_id` is ignored; retention job and dashboard show drop/allow counts without PII. |
| SEC-07 | `image-proxy` checks textual private hosts but does not resolve DNS before connecting. DNS rebinding and IPv6/private resolution remain SSRF paths. | Replace ad hoc host filtering with DNS resolution plus address-class validation for every resolved A/AAAA address, block redirects, restrict methods/content types/sizes, and cache safe resolutions briefly. Prefer signed Storage transformations for internal assets. | Tests cover loopback, link-local, RFC1918, CGNAT, IPv6 local ranges, redirect chains, and rebinding simulation. |
| SEC-08 | Corporate email sanitization exists in browser flows, but the direct sending path accepts raw `bodyHtml`. A future caller can bypass DOMPurify. | Sanitize again in the Edge Function with a server-compatible allowlist; remove dangerous URL schemes, event attributes, forms, style injection, and remote image tracking unless approved; record only a content hash in logs. | Malicious HTML fixtures render harmlessly; every mail entrypoint uses the shared sanitizer; no recipient/email body is logged in clear text. |
| SEC-09 | Edge logs include recipient email addresses, push/device tokens, signed URLs, and payload fragments in several locations. This increases breach and support-log exposure. | Define a structured logging policy: correlation id, event type, user UUID hash, provider status, and safe counts only. Use the shared redactor at every log boundary; set retention/access policy in Supabase/Cloudflare. | A log-fixture test asserts secrets/PII are redacted; runtime sampling confirms useful diagnostics remain. |
| SEC-10 | `public/_headers` uses report-only CSP and permits `unsafe-inline`/`unsafe-eval`. Report-only cannot contain an XSS incident. | Inventory script/style/eval requirements, remove eval-capable dependencies/configurations, migrate remaining inline bootstraps to hashes/nonces, then deploy enforced CSP with report endpoint and a staged fallback policy. | No violation regressions on authenticated and public flows for seven days; header scan reports enforcement; CSP report volume is reviewed. |
| SEC-11 | Public/tokenized/anonymous Edge Functions and 82 anonymous/PUBLIC function grants are governed by a baseline but require semantic review. | Build a manifest declaring each function/RPC's audience, token type, rate limit, input size, data classification, and owner. Revoke unneeded grants; move public workflows to purpose-specific signed tokens with expiry and audience checks. | Manifest is generated/validated in CI; no unclassified public function/RPC; negative auth tests cover each public entrypoint. |
| DATA-01 | Transport request creation has a read-then-create duplicate check, so concurrent calls can still create duplicates; parent/item writes are not atomic at the database level. | Add a backward-compatible idempotency column or a dedicated `subrental_id` relation plus unique constraint; implement an RLS-checked transactional RPC that inserts the request and items together. Migrate existing note markers gradually. | Parallel invocation test produces one request; retry returns the original record; database test proves atomic rollback. |
| DATA-02 | The primary app code is typechecked, but Edge Functions are only linted and use separate Deno imports. Privileged code can regress without a type gate. | Add Deno type-check/test commands per function package or a Supabase Functions CI matrix. Make the shared modules testable without Node aliases (as in this PR). | CI typechecks all changed functions; changes to `_shared` run focused tests; deployment smoke tests are documented. |

## Ownership and review rules

- A roadmap card has one accountable product/engineering owner, a reviewer familiar with the affected domain, and an explicit target date.
- Security/RLS changes require a second reviewer and negative authorization tests. Do not merge a policy change based on static text alone.
- No “AI cleanup” bulk PRs: preserve behavior with focused modules, measurable baseline changes, and deletion of obsolete compatibility code only after callers move.
- Every exception is temporary: record why, who owns it, and its expiry. A baseline is a transition mechanism, not acceptance of permanent debt.
